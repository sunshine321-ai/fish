# 第四章 系统实现

本章在第三章系统设计的基础上，对MiroFish多智能体仿真预测系统的具体实现过程进行说明。根据系统实现阶段的主要任务，本章分为编码实现和系统测试两部分。编码实现部分主要介绍系统开发环境、后端服务、实际用户界面、核心业务流程、数据存储和外部服务调用的实现方式；系统测试部分主要介绍测试基础、测试方法、单元测试、集成测试、验收测试以及调试与可靠性分析。按照课程要求，测试部分不列出全部测试用例，而是分别选取单元测试、集成测试和验收测试中的一个代表性用例进行说明。

## 4.1 编码实现

### 4.1.1 开发环境与编码规范

MiroFish系统采用前后端分离的开发方式。前端负责用户界面展示、交互状态维护和异步接口调用，后端负责业务逻辑处理、文件管理、知识图谱构建、多智能体仿真和报告生成。系统主要开发环境如表4-1所示。

**表4-1 系统开发环境**

| 类型 | 技术或工具 | 说明 |
|------|------------|------|
| 操作系统 | Windows | 本地开发与调试环境 |
| 后端语言 | Python 3.11及以上 | 实现后端接口、业务逻辑和仿真调度 |
| 后端框架 | Flask | 提供REST API服务 |
| 前端框架 | Vue 3 | 构建单页应用界面 |
| 构建工具 | Vite | 前端开发与构建工具 |
| HTTP客户端 | Axios | 前端调用后端接口 |
| 图谱可视化 | D3.js | 前端图谱节点与关系可视化 |
| 大语言模型接口 | OpenAI SDK兼容接口 | 用于本体生成、人设生成和报告生成 |
| 知识图谱服务 | Zep Cloud | 用于图谱存储、实体查询和关系检索 |
| 仿真框架 | CAMEL OASIS | 用于Twitter和Reddit双平台多智能体仿真 |
| 包管理工具 | npm、uv | 管理前端和后端依赖 |

系统代码按照前后端分离结构组织。后端代码位于`backend`目录下，其中`app/api`目录存放REST接口，`app/services`目录存放核心业务服务，`app/models`目录存放项目、任务等数据模型，`app/utils`目录存放文件解析、日志、LLM客户端等工具模块。前端代码位于`frontend/src`目录下，其中`views`目录存放页面级组件，`components`目录存放流程步骤和图谱面板等复用组件，`api`目录对后端接口进行封装。

在编码规范方面，后端采用模块化组织方式，将不同业务能力拆分为图谱、仿真和报告三个接口蓝图；对外部配置统一从`.env`文件读取，并在系统启动前完成必要配置校验；对较长耗时任务采用任务状态或日志文件进行进度记录。前端采用Vue单文件组件组织界面，将页面路由、接口请求和状态数据分离，保证界面展示逻辑与后端业务逻辑之间边界清晰。

### 4.1.2 后端服务实现

后端服务以`backend/run.py`作为启动入口。系统启动时首先调用`Config.validate()`函数检查必要配置项是否完整，然后通过`create_app()`函数创建Flask应用实例，并根据环境变量读取主机、端口和调试模式，最后启动HTTP服务。

Flask应用创建逻辑位于`backend/app/__init__.py`。该文件采用应用工厂模式实现`create_app()`函数，在应用初始化过程中完成配置加载、跨域支持、请求日志中间件注册和蓝图注册。系统后端主要蓝图如表4-2所示。

**表4-2 后端蓝图与接口职责**

| 蓝图 | 注册路径 | 主要职责 | 代表函数 |
|------|----------|----------|----------|
| `graph_bp` | `/api/graph` | 项目管理、本体生成、图谱构建、图谱查询 | `generate_ontology()`、`build_graph()`、`get_graph_data()` |
| `simulation_bp` | `/api/simulation` | 仿真实例创建、人设生成、仿真准备、仿真运行与状态查询 | `create_simulation()`、`prepare_simulation()`、`start_simulation()` |
| `report_bp` | `/api/report` | 报告生成、报告查询、报告对话和图谱工具调用 | `generate_report()`、`get_report()`、`chat_with_report_agent()` |

图谱相关接口位于`backend/app/api/graph.py`。系统首先通过`generate_ontology()`接口接收用户上传的资料和仿真需求，调用`ProjectManager.create_project()`创建项目目录，调用`ProjectManager.save_file_to_project()`保存上传文件，并通过`FileParser.extract_text()`和`TextProcessor.preprocess_text()`完成文本抽取和预处理。随后系统调用`OntologyGenerator.generate()`生成实体类型、关系类型和分析摘要，并将结果保存到项目元数据中。

图谱构建由`build_graph()`接口负责。该接口读取项目中保存的抽取文本，使用`TextProcessor.split_text()`按照设定的分块大小和重叠长度对文本进行切分，然后创建异步任务并调用`GraphBuilderService`完成Zep知识图谱构建。构建完成后，系统将`graph_id`写入项目元数据，前端可通过`get_graph_data()`接口读取图谱节点和关系数据。

仿真相关接口位于`backend/app/api/simulation.py`。`create_simulation()`函数根据项目ID和图谱ID创建仿真实例，并由`SimulationManager.create_simulation()`保存仿真状态。`prepare_simulation()`函数负责仿真环境准备，包括读取图谱实体、生成Agent人设、生成双平台仿真配置和保存配置文件。仿真运行由`start_simulation()`函数触发，该函数调用`SimulationRunner.start_simulation()`启动仿真进程，并通过`get_run_status()`、`get_run_status_detail()`、`get_simulation_timeline()`等接口向前端返回运行状态、动作流水和时间线数据。

报告相关接口位于`backend/app/api/report.py`。`generate_report()`函数根据仿真ID、图谱ID和仿真需求创建`ReportAgent`，并调用其`generate_report()`方法逐章生成预测报告。报告生成过程中，系统通过`ReportManager.update_progress()`、`ReportManager.save_section()`等函数保存报告进度和章节内容。报告完成后，前端可通过`get_report()`、`get_report_progress()`、`get_report_sections()`等接口加载报告正文、章节和进度信息；用户也可以通过`chat_with_report_agent()`函数与报告智能体进行交互式问答。

### 4.1.4 核心业务流程实现

MiroFish系统的核心业务流程为“资料上传—本体生成—图谱构建—环境搭建—仿真运行—报告生成与交互”。各阶段之间以前一阶段的输出作为后一阶段输入，形成完整的仿真预测闭环。

第一阶段是资料上传与项目创建。用户在首页上传资料并填写预测需求后，前端进入流程页并调用`generateOntology()`接口。后端`generate_ontology()`函数创建项目对象，保存用户上传文件，并将文件内容解析为文本。项目元数据保存在`uploads/projects/{project_id}/project.json`中，上传文件保存在项目目录下的`files`子目录中，提取后的文本保存在`extracted_text.txt`中。

第二阶段是本体生成。系统将上传资料中的文本和用户预测需求作为上下文，调用`OntologyGenerator.generate()`函数生成实体类型、关系类型和分析摘要。本体结果保存到项目元数据中，并作为后续知识图谱构建的约束条件。

第三阶段是知识图谱构建。后端`build_graph()`函数读取项目文本和本体信息，使用`TextProcessor.split_text()`将长文本划分为多个片段，并通过`GraphBuilderService`调用Zep服务写入图谱数据。该过程通过`TaskManager`维护异步任务状态，前端可通过`getTaskStatus()`轮询任务进度，并在构建完成后通过`getGraphData()`加载节点与关系数据。

第四阶段是仿真环境搭建。用户进入环境搭建界面后，系统调用`create_simulation()`创建仿真实例，再调用`prepare_simulation()`完成准备工作。准备过程包括读取图谱实体、调用`OasisProfileGenerator.generate_profiles_from_entities()`生成Agent人设、调用`SimulationConfigGenerator.generate_config()`生成时间配置、事件配置和平台参数，并将结果保存到仿真目录中。

第五阶段是多智能体仿真运行。用户确认参数后调用`start_simulation()`接口，后端通过`SimulationRunner.start_simulation()`启动Twitter和Reddit双平台仿真进程。仿真运行过程中，系统持续读取动作日志、帖子、评论和轮次状态，并通过`get_run_status()`、`get_simulation_actions()`、`get_simulation_timeline()`等接口提供给前端展示。

第六阶段是报告生成与交互。仿真完成后，用户进入报告界面并触发`generate_report()`接口。后端创建`ReportAgent`，结合图谱查询、统计分析和仿真结果逐章生成预测报告，报告章节、进度、日志和最终Markdown文件由`ReportManager`统一保存。报告生成后，用户还可以继续向Report Agent提问，或向仿真世界中的Agent发起访谈，从而完成对预测结果的进一步分析和验证。

### 4.1.5 数据存储与外部服务调用实现

系统采用本地文件系统和外部云服务结合的方式进行数据管理。本地文件系统主要保存项目元数据、上传文件、提取文本、仿真配置、仿真日志、报告章节和报告正文；外部Zep服务主要保存知识图谱节点、边、实体上下文和长期记忆；LLM服务主要承担文本理解、结构化生成、Agent人设生成和报告推理任务。

本地数据目录主要位于`backend/uploads`下，其结构如表4-4所示。

**表4-4 本地数据存储结构**

| 目录 | 数据内容 | 说明 |
|------|----------|------|
| `uploads/projects` | 项目元数据、上传文件、抽取文本 | 保存项目级数据 |
| `uploads/simulations` | 仿真实例、Agent人设、仿真配置、运行日志 | 保存仿真过程数据 |
| `uploads/reports` | 报告元数据、章节内容、Markdown正文、Agent日志 | 保存报告生成结果 |

外部服务调用由服务层统一封装。LLM相关能力由`LLMClient`、`OntologyGenerator`、`OasisProfileGenerator`、`SimulationConfigGenerator`和`ReportAgent`等模块调用。Zep相关能力由`GraphBuilderService`、`ZepEntityReader`、`ZepToolsService`和`ZepGraphMemoryUpdater`等模块调用。前端不直接接触外部服务密钥，所有密钥均由后端从`.env`文件读取，减少敏感信息暴露风险。

## 4.2 系统测试

### 4.2.1 软件测试基础

软件测试的目标是发现系统在功能实现、接口调用、数据处理和运行稳定性方面存在的问题，验证系统是否满足需求规格说明书中的功能要求。MiroFish系统涉及前端页面、后端接口、文件存储、外部LLM服务、Zep知识图谱服务和多智能体仿真进程，测试过程中需要同时关注单个函数的正确性、模块之间的数据传递以及完整用户流程是否可用。

本系统测试遵循以下原则：第一，测试内容与需求功能保持一致，重点验证资料上传、图谱构建、Agent生成、仿真运行和报告生成等核心功能；第二，测试对象从函数、接口逐步扩展到完整业务流程，避免只验证最终页面而忽略中间数据状态；第三，测试结果以实际输入、预期输出和测试结论为依据，确保测试过程可复现。

### 4.2.2 测试方法

从测试层次看，本系统测试分为单元测试、集成测试和验收测试。单元测试用于验证单个函数或模块的内部逻辑是否正确；集成测试用于验证多个模块组合后的接口调用和数据传递是否正确；验收测试用于从用户需求角度验证系统是否满足完整业务流程。

从测试方法看，本系统采用白盒测试与黑盒测试相结合的方式。白盒测试关注程序内部逻辑，主要用于单元测试，例如配置校验、参数校验和分支处理逻辑；黑盒测试关注输入和输出，主要用于集成测试和验收测试，例如前端操作后接口返回是否正确、完整流程是否能输出预测报告。

在集成测试策略方面，本系统采用渐增式集成测试中的自顶向下策略。由于MiroFish系统具有明显的用户操作顺序，即“资料上传—图谱构建—环境搭建—仿真运行—报告生成—深度交互”，因此从前端页面和后端接口入口出发，逐步验证业务服务、文件存储和外部服务之间的协作关系，更便于定位流程衔接中的问题。

### 4.2.3 单元测试

单元测试主要用于验证系统中相对独立的函数逻辑。单元测试通常需要编写专门的测试函数，通过给被测函数提供特定输入并检查返回结果，判断其内部逻辑是否符合预期。本系统选择`Config.validate()`函数作为被测函数。该函数位于`backend/app/config.py`，用于在系统启动前检查必要环境变量是否完整。由于该函数内部包含对`LLM_API_KEY`和`ZEP_API_KEY`的配置分支判断，适合采用白盒测试方法进行验证。

**表4-5 单元测试用例**

| 项目 | 内容 |
|------|------|
| 测试编号 | UT-01 |
| 测试对象 | `Config.validate()`函数 |
| 函数位置 | `backend/app/config.py` |
| 测试方法 | 白盒测试 |
| 测试内容 | 验证系统启动前配置校验逻辑是否正确 |
| 输入数据 | 缺少`LLM_API_KEY`或`ZEP_API_KEY`的`.env`配置 |
| 测试步骤 | 修改测试环境中的配置项；调用`Config.validate()`函数；观察返回的错误列表 |
| 预期结果 | 函数返回配置错误信息，`run.py`根据错误结果终止系统启动 |
| 实际结果 | 函数能够识别缺失配置项并返回错误列表 |
| 测试结论 | 通过 |

针对该测试对象，可以编写如下测试函数：

```python
def test_config_validate_missing_required_keys(monkeypatch):
    monkeypatch.setattr(Config, "LLM_API_KEY", None)
    monkeypatch.setattr(Config, "ZEP_API_KEY", None)

    errors = Config.validate()

    assert "LLM_API_KEY 未配置" in errors
    assert "ZEP_API_KEY 未配置" in errors
```

该测试函数通过临时修改`Config`类中的配置值，模拟系统缺少必要环境变量的情况，然后调用`Config.validate()`函数并检查返回结果中是否包含对应错误信息。通过该用例可以验证系统在缺少关键外部服务配置时能够提前发现问题，避免后续运行阶段才出现LLM或Zep服务调用失败。

### 4.2.4 集成测试

集成测试用于验证多个模块组合后的协作关系。本系统选择环境准备、仿真启动和运行状态查询流程作为集成测试对象。该流程涉及前端环境搭建界面、后端仿真接口、Agent人设生成服务、仿真配置生成服务、仿真运行服务和文件存储模块，能够较好地体现模块之间的数据传递关系。

### 4.2.6 调试与可靠性分析

在系统实现和测试过程中，调试主要围绕前端页面、后端接口、外部服务调用和仿真进程四类问题展开。前端调试主要通过浏览器控制台和接口响应信息定位页面状态、路由跳转和请求参数问题；后端调试主要通过请求日志、异常日志和接口返回状态定位参数缺失、文件读取失败和业务处理异常；外部服务调试主要检查`.env`配置、API密钥、网络连接和服务返回内容；仿真进程调试主要通过运行状态文件、动作日志、控制台日志和IPC状态判断仿真是否正常运行。

系统可靠性主要通过以下方式保障。第一，启动阶段通过`Config.validate()`函数校验关键配置，避免配置缺失导致运行时错误。第二，后端接口对关键参数进行检查，例如项目ID、图谱ID、仿真ID和报告ID缺失时返回错误信息。第三，系统将项目、仿真和报告状态持久化到本地文件中，即使前端刷新页面，也可以通过接口重新加载当前进度。第四，耗时任务采用任务状态、进度文件或日志文件记录执行过程，便于前端轮询和问题定位。第五，外部服务调用被封装在服务层中，减少外部服务变化对前端和上层业务逻辑的影响。

总体来看，MiroFish系统通过配置校验、异常处理、状态持久化、日志记录和模块化接口设计，提高了系统在多步骤仿真预测流程中的可调试性和运行可靠性。
